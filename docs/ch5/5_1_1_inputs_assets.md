# 5.1.1 Inputs/Assets

Η υλοποίηση του OG–NSD ξεκινά από ένα σύνολο εισόδων (inputs/assets) που είναι πλήρως εμφανές στο αποθετήριο και επιτρέπει αναπαραγωγιμότητα. Κεντρικό ρόλο παίζουν τα αρχεία απαιτήσεων σε JSONL, π.χ. `atm_requirements.jsonl`, `baseline_requirements.jsonl`, και `health_requirements.jsonl`. Κάθε γραμμή αντιστοιχεί σε μία απαίτηση φυσικής γλώσσας και τροφοδοτείται στον loader (`og_nsd/requirements.py`), ο οποίος χειρίζεται parsing, καθαρισμό whitespace και ενδεχόμενο chunking. Η επιλογή JSONL επιτρέπει ροή επεξεργασίας και στοχευμένο υποσύνολο μέσω `--max-reqs`, χωρίς να απαιτείται φόρτωση όλου του corpus στη μνήμη.

Το δεύτερο κρίσιμο asset είναι τα gold ontologies και τα SHACL shapes στο `gold/`. Για το ATM domain, το `gold/atm_gold.ttl` παρέχει το λεξιλόγιο (κλάσεις/ιδιότητες/συμβάσεις ονοματοδοσίας) ενώ το `gold/shapes_atm.ttl` κωδικοποιεί τις δομικές και τυπολογικές δεσμεύσεις. Αντίστοιχα, για healthcare υπάρχουν `gold/health_gold.ttl` και `gold/shapes_health.ttl`. Αυτά τα αρχεία χρησιμοποιούνται ως grounding κατά το ontology-aware prompting (χωρίς να αντιγράφονται αυτούσιες οι αξιωματικές δηλώσεις) και ως κανόνες επικύρωσης μέσω SHACL. Το pipeline δέχεται επίσης προαιρετικό `--base` αρχείο, το οποίο συγχωνεύεται με το παραγόμενο γράφημα ώστε να δημιουργηθεί ενιαία οντολογία.

Τρίτο είδος asset είναι τα competency questions (CQs), αποθηκευμένα ως SPARQL ASK queries σε `.rq` αρχεία, π.χ. `atm_cqs.rq` και `health_cqs.rq`. Η αξιολόγηση CQs γίνεται από το module `og_nsd/queries.py` και λειτουργεί ως ανεξάρτητη μέτρηση επάρκειας του παραγόμενου μοντέλου γνώσης. Οι CQ έλεγχοι ενσωματώνονται στο report ανά iteration, κάτι που επιτρέπει την ανάλυση της προόδου του repair loop.

Ένα επιπλέον asset είναι οι ρυθμίσεις εκτέλεσης. Τα JSON config αρχεία στον φάκελο `configs/` (π.χ. `configs/atm_e4_iterative.json`) αποτυπώνουν όλες τις παραμέτρους του pipeline: LLM mode, αριθμό iterations, paths για requirements, shapes, base και CQs. Η χρήση config επιτρέπει τυποποιημένη αναπαραγωγή πειραμάτων (E1–E6) και οργανώνει πολλαπλά domains χωρίς αλλαγή κώδικα.

Τέλος, τα outputs που αποθηκεύονται στο `runs/` θεωρούνται επίσης “assets” για την επιστημονική τεκμηρίωση. Κάθε experiment script (π.χ. `scripts/run_e4_iterative.py`) γράφει Turtle αρχεία, αναφορές JSON, καθώς και ενδιάμεσα artefacts, ώστε να καταγράφονται οι επιπτώσεις των αποφάσεων του LLM και της symbolic επικύρωσης. Το `build/` συγκεντρώνει ad-hoc outputs από τις βασικές εκτελέσεις του pipeline. Η συστηματική οργάνωση των εισόδων/outputs καθιστά σαφές ποια δεδομένα τροφοδοτούνται στο σύστημα και ποια παράγονται, κάτι κρίσιμο για ακαδημαϊκή τεκμηρίωση.
