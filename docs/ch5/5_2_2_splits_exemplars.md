# 5.2.2 Splits & Exemplars

Στο OG–NSD, η έννοια των “splits” δεν αφορά παραδοσιακή εποπτευόμενη μάθηση με train/val/test πάνω σε labeled data, αλλά την οργάνωση των απαιτήσεων και των πειραμάτων ώστε να υπάρχει επαναληψιμότητα και σύγκριση μεταξύ μεθόδων. Το `--max-reqs` στο CLI και οι αντίστοιχες παράμετροι στα config αρχεία επιτρέπουν την επιλογή υποσυνόλων απαιτήσεων. Αυτό χρησιμοποιείται τόσο για γρήγορα pilot runs όσο και για συγκεκριμένα experiment protocols (π.χ. να χρησιμοποιούνται οι ίδιες 50 απαιτήσεις σε όλα τα experiments E1–E4). Το pipeline κρατά σταθερή σειρά με βάση την ανάγνωση από JSONL, ώστε οι συγκρίσεις να είναι δίκαιες.

Η υλοποίηση επιτρέπει επίσης chunking απαιτήσεων μέσω `RequirementLoader.chunk_requirements`, κάτι που λειτουργεί ως “split” σε μικρότερα batches για prompt construction. Αυτό είναι σημαντικό διότι τα LLMs έχουν περιορισμένο context window. Έτσι, ένα μεγάλο σύνολο απαιτήσεων μπορεί να χωριστεί σε groups, και το LLM να παράγει τμηματικά axioms. Τα παραγόμενα snippets συγχωνεύονται αργότερα στο τελικό ontology. Αυτή η στρατηγική είναι χρήσιμη σε domains με μεγάλα requirement sets και επιτρέπει την κλιμάκωση χωρίς να αλλοιωθεί η λογική της pipeline.

Η χρήση exemplars (few-shot παραδείγματα) αποτελεί επίσης μέρος της προεπεξεργασίας. Παρότι τα exemplars δεν είναι ξεχωριστά αρχεία στο repo, η λογική τους είναι ενσωματωμένη στο prompt construction στο `og_nsd/llm.py`. Το README περιγράφει ότι τα exemplars είναι μικρά snippets Turtle με σχόλια που χαρτογραφούν φράσεις απαιτήσεων σε OWL axioms. Αυτά τα παραδείγματα λειτουργούν ως “style guide” για το LLM: δείχνουν την αναμενόμενη σύνταξη, τη χρήση namespaces και το πώς να εκφράζονται domain/range constraints. Σε ontology-aware mode, τα exemplars συνδυάζονται με το extracted vocabulary context για να περιορίσουν περαιτέρω το drift.

Από πλευράς πειραματικής αξιολόγησης, τα splits μεταξύ domains είναι βασικό στοιχείο. Τα config αρχεία του E5 (cross-domain) περιλαμβάνουν πολλαπλά domain blocks (ATM και health), όπου κάθε block ορίζει διαφορετικά requirements/shapes/base/CQs. Αυτή η δομή λειτουργεί ως διαχωρισμός dataset σε “task splits” και επιτρέπει να μετρηθεί η γενίκευση του pipeline από ένα domain στο άλλο χωρίς αλλαγή κώδικα.

Η ακαδημαϊκή τεκμηρίωση πρέπει να τονίζει ότι η επιλογή splits και exemplars δεν είναι μεθοδολογικά τυχαία. Τα splits καθορίζουν τη συγκρισιμότητα των πειραμάτων, ενώ τα exemplars εξασφαλίζουν ότι το LLM ακολουθεί σταθερό format και δεν εισάγει συντακτικά σφάλματα. Τα δύο αυτά στοιχεία λειτουργούν ως “πειθαρχητική” υποδομή: μειώνουν το noise στα αποτελέσματα και επιτρέπουν την καθαρή αποτίμηση της συμβολής του repair loop και της symbolic επικύρωσης.
